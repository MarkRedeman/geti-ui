## Responsibility

Provides the complete toast notification system: a mount-point component (`ToastContainer`) that renders queued toasts in the DOM, and a programmatic imperative API (`toast`) for enqueuing notifications from anywhere in the application without needing component context.

## When to use

- **Success Confirmation**: To notify a user that an action (e.g., saving a project, deleting an item) completed successfully.
- **Error Alerts**: To warn a user about a failed process or network error that doesn't block the entire workflow.
- **Informational Updates**: To provide non-critical context about background processes.
- **Async Feedback**: To confirm when a long-running background task has finished.

## When to avoid

- **Critical Warnings**: If a user must acknowledge a message before proceeding, use a `Dialog` or `AlertDialog` instead.
- **Lengthy Content**: Toasts should be concise. For detailed logs or long messages, use a dedicated notification center or status page.
- **Persistent Information**: Toasts disappear automatically; do not use them for information the user needs to refer back to later.

## Design

Two exports with different usage models:

**`ToastContainer`** тАФ thin wrapper over Spectrum's `ToastContainer`. Must be rendered once in the application tree (typically near the root). Accepts the same props as `SpectrumToastContainer`.

**`toast`** тАФ a plain object (not a component) with four methods:
```ts
toast.positive(message, options?)
toast.negative(message, options?)
toast.info(message, options?)
toast.neutral(message, options?)
```
Each method calls `SpectrumToastQueue.<variant>(message, options)` directly. This gives callers a stable, variant-named API without importing Spectrum internals. The `toast` object is a pure function dispatch table тАФ no hooks, no React lifecycle.

The separation of `ToastContainer` (render tree) from `toast` (imperative API) follows the "render once, call anywhere" pattern, enabling toast creation from event handlers, async callbacks, and non-React code.

## Flow

**Setup (once):**
```
<ToastContainer /> mounted in app root
  тЖТ renders SpectrumToastContainer (manages its own queue)
```

**Usage (anywhere):**
```
toast.negative('Upload failed', { timeout: 5000 })
  тЖТ SpectrumToastQueue.negative('Upload failed', { timeout: 5000 })
  тЖТ ToastContainer re-renders with new toast
```

## Integration

- `ToastContainer` must be present in the React tree for toasts to appear.
- `toast.*` can be called from any module тАФ event handlers, async utilities, error boundaries тАФ without component imports.
- Spectrum manages toast queuing, deduplication, and auto-dismiss internally.

## Accessibility

Toasts are rendered in an ARIA live region to ensure they are announced by screen readers immediately upon appearing.

| Feature | Implementation |
| :--- | :--- |
| **Role** | Uses `role="status"` or `role="alert"` depending on the variant (negative uses alert). |
| **Live Region** | `aria-live="polite"` for most toasts; `aria-live="assertive"` for critical errors. |
| **Focus** | Toasts do not take focus on appearance to avoid interrupting the user's flow. |
| **Keyboard** | Users can navigate to active toasts via `F6` (standard Spectrum behavior) to interact with action buttons. |

### ToastOptions
| Property | Type | Default | Description |
| :--- | :--- | :--- | :--- |
| `timeout` | `number` | `5000` | Time in milliseconds before the toast automatically dismisses. |
| `actionLabel` | `string` | - | Label for an optional action button within the toast. |
| `onAction` | `() => void` | - | Callback triggered when the action button is clicked. |
| `shouldCloseOnAction` | `boolean` | `true` | Whether the toast should dismiss automatically after the action is triggered. |
