# Dialog

Dialogs are windows containing contextual information or tasks that appear over the main UI content.

```tsx
import { Dialog, DialogTrigger } from '@geti-ai/ui';
```

- [React Spectrum V3 Dialog](https://react-spectrum.adobe.com/react-spectrum/Dialog.html)

## Responsibility

`Dialog` is a **content container** â€” it defines the interior structure and slots of a modal window. On its own it does not open, close, position itself, or manage focus. Those responsibilities belong to `DialogTrigger`, which acts as the host and orchestrates the full overlay lifecycle.

The two components are always used together:

```tsx
<DialogTrigger>
    {/* 1. The element that opens the dialog */}
    <Button variant="accent">Open dialog</Button>

    {/* 2. A function (or element) that returns the dialog content */}
    {(close) => (
        <Dialog>
            <Heading>Confirm deletion</Heading>
            <Divider />
            <Content>This action cannot be undone.</Content>
            <Footer>
                <ButtonGroup>
                    <Button variant="secondary" onPress={close}>
                        Cancel
                    </Button>
                    <Button variant="negative" onPress={close}>
                        Delete
                    </Button>
                </ButtonGroup>
            </Footer>
        </Dialog>
    )}
</DialogTrigger>
```

Together, `Dialog` + `DialogTrigger` handle:

- **Focus trapping**: Focus is contained within the dialog while it is open. Tab cycles through all focusable children; Shift+Tab reverses direction.
- **Focus restoration**: When the dialog closes, focus returns to the element that triggered it.
- **Overlay rendering**: The underlay (modal backdrop) is rendered and managed by `DialogTrigger`, not `Dialog`.
- **Dismissal**: Clicking the underlay or pressing `Escape` closes the dialog (unless disabled via `isKeyboardDismissDisabled`).
- **ARIA wiring**: Spectrum links `Dialog`'s `Heading` slot to the dialog element via `aria-labelledby` automatically.

### When to use

- To prompt the user for a critical decision or confirmation before a destructive or irreversible action (e.g., "Are you sure you want to delete this project?").
- To gather structured input via a form without navigating away from the current context (e.g., "Create new dataset").
- To surface detailed information about an item that would clutter the main view if shown inline.
- When blocking the underlying content is intentional â€” the user must respond before continuing.

### When to avoid

- For transient feedback or status notifications â€” use `Toast` instead.
- For brief, non-interactive hints or explanations â€” use `Tooltip` or `ContextualHelp`.
- For persistent side-panels or drawers â€” use a dedicated side-panel layout component rather than a fullscreen dialog.
- When the action has low consequence and does not require explicit confirmation â€” prefer inline affordances to reduce interruption.

## Design

`Dialog` follows the _transparent wrapper_ pattern: it extends `SpectrumDialogProps` without modification and delegates all rendering to Spectrum.

```tsx
// packages/ui/src/components/overlays/Dialog/Dialog.tsx

export interface DialogProps extends SpectrumDialogProps {}

export const Dialog = (props: DialogProps) => <SpectrumDialog {...props} />;
```

`DialogTrigger` follows the same pattern:

```tsx
// packages/ui/src/components/overlays/Dialog/DialogTrigger.tsx

export interface DialogTriggerProps extends SpectrumDialogTriggerProps {}

export const DialogTrigger = (props: DialogTriggerProps) => <SpectrumDialogTrigger {...props} />;
```

This design keeps the Geti wrappers as stable integration points â€” they can later adopt custom behaviour (e.g., analytics, animation overrides, or migration to `react-aria-components`) without changing the consumer API.

**Slot-based content model**

`Dialog` uses a slot system inherited from Spectrum. Each named slot renders in a specific position and receives predefined styles:

| Slot component  | Position           | Required?                                        |
| --------------- | ------------------ | ------------------------------------------------ |
| `<Heading>`     | Dialog title bar   | Strongly recommended (used as `aria-labelledby`) |
| `<Divider>`     | Below the title    | Optional                                         |
| `<Content>`     | Main body          | Recommended for body text or forms               |
| `<Footer>`      | Bottom, full-width | Optional; use for secondary info                 |
| `<ButtonGroup>` | Bottom-right       | Recommended for action buttons                   |

**Theme integration**

`Dialog` inherits all Spectrum design tokens (overlay colour, border radius, shadow) from the `ThemeProvider` context. No per-component CSS overrides are needed in standard usage.

## Variants

### Modal (default)

The default overlay type. An underlay blocks interaction with the rest of the application until the dialog is dismissed. Use for decisions, forms, and confirmations.

```tsx
<DialogTrigger>
    {' '}
    {/* type defaults to 'modal' */}
    <Button>Open</Button>
    <Dialog>...</Dialog>
</DialogTrigger>
```

### Popover

A smaller dialog anchored relative to its trigger. Does not block interaction with the rest of the UI and dismisses on outside click. Use for contextual help or quick-edit panels.

```tsx
<DialogTrigger type="popover">
    <Button>Settings</Button>
    <Dialog>...</Dialog>
</DialogTrigger>
```

### Fullscreen

Occupies the entire viewport. Use for complex multi-step tasks or dense data-entry forms that benefit from maximum space.

```tsx
<DialogTrigger type="fullscreen">
    <Button>Open editor</Button>
    <Dialog>...</Dialog>
</DialogTrigger>
```

### Dismissible

Renders a close ("Ă—") button in the corner of the dialog, giving users a clear escape path. Enabled via the `isDismissable` prop on `DialogTrigger` (not on `Dialog` itself).

```tsx
<DialogTrigger isDismissable>
    <Button>Open</Button>
    <Dialog>...</Dialog>
</DialogTrigger>
```

### Sizes

`Dialog` accepts a `size` prop that controls its maximum width. The medium size (`M`) is the default and appropriate for most use cases.

| Size          | Use Case                                           |
| ------------- | -------------------------------------------------- |
| `S`           | Simple confirmations with minimal content          |
| `M` (default) | Standard decisions, short forms                    |
| `L`           | Rich content, multi-field forms, or embedded media |

## Props

### Dialog Props

| Prop            | Type                                            | Default | Description                                                                                                          |
| --------------- | ----------------------------------------------- | ------- | -------------------------------------------------------------------------------------------------------------------- |
| `children`      | `ReactNode \| (close: () => void) => ReactNode` | â€”       | The contents of the dialog. Supports slot-based children (`Heading`, `Divider`, `Content`, `ButtonGroup`, `Footer`). |
| `size`          | `'S' \| 'M' \| 'L'`                             | `'M'`   | Controls the maximum width of the dialog.                                                                            |
| `isDismissable` | `boolean`                                       | `false` | Whether to render a close (Ă—) button in the corner of the dialog.                                                    |

| Prop                        | Type                                                                     | Default   | Description                                                                                   |
| --------------------------- | ------------------------------------------------------------------------ | --------- | --------------------------------------------------------------------------------------------- |
| `type`                      | `'modal' \| 'popover' \| 'tray' \| 'fullscreen' \| 'fullscreenTakeover'` | `'modal'` | The type of overlay to render.                                                                |
| `mobileType`                | `'modal' \| 'tray' \| 'fullscreen'`                                      | `'tray'`  | The overlay type used on mobile viewports.                                                    |
| `isOpen`                    | `boolean`                                                                | â€”         | Whether the dialog is open (controlled mode).                                                 |
| `defaultOpen`               | `boolean`                                                                | `false`   | Whether the dialog is open initially (uncontrolled mode).                                     |
| `onOpenChange`              | `(isOpen: boolean) => void`                                              | â€”         | Called when the open state changes.                                                           |

## Accessibility

- **ARIA role**: Spectrum assigns `role="dialog"` to the dialog element. For critical alerts that require an immediate response, you can use `role="alertdialog"` by customising the underlying Spectrum component.
- **Labeling via slot**: The `<Heading>` slot is automatically linked to the dialog via `aria-labelledby`. Always include a `<Heading>` that clearly describes the purpose of the dialog. If no heading is visible, add `aria-label` directly to `<Dialog>`.
- **Describing content**: If the dialog contains supplementary information beyond the heading, place it in a `<Content>` slot â€” Spectrum may wire it as `aria-describedby` where appropriate.
- **Focus trap**: While the dialog is open, focus is trapped within it. All interactive elements (buttons, inputs, links) within the dialog remain reachable via `Tab`. Content behind the dialog is inert.
- **Keyboard interactions**:

| Key               | Behaviour                                                    |
| ----------------- | ------------------------------------------------------------ |
| `Tab`             | Move focus to the next focusable element within the dialog   |
| `Shift + Tab`     | Move focus to the previous focusable element                 |
| `Escape`          | Close the dialog (unless `isKeyboardDismissDisabled` is set) |
| `Enter` / `Space` | Activate the focused button                                  |

- **Focus restoration**: When the dialog closes, focus automatically returns to the element that triggered it â€” typically the `Button` that opened the dialog. No manual focus management is required.
- **Screen reader announcements**: When the dialog opens, screen readers announce its title (from `<Heading>`) and role. The `role="dialog"` landmark ensures users can navigate to it via their reader's landmark navigation shortcuts.

## Integration

- **Dependencies**: `@adobe/react-spectrum` â€” `Dialog`, `DialogTrigger`, and their Spectrum prop types are imported from this package.
- **Slot components**: `Heading`, `Divider`, `Content`, `Footer`, `ButtonGroup` are imported from `@adobe/react-spectrum`. They are not re-exported by `@geti-ai/ui` at this time â€” import them directly.
- **ThemeProvider**: Must be present in the React tree. The overlay background, border radius, and shadow all derive from Spectrum tokens resolved by `ThemeProvider`.
- **Button**: Use `@geti-ai/ui`'s `Button` for all action buttons inside a `Dialog`. This ensures consistent `accent` defaults and correct focus ref types.
- **z-index and stacking**: Spectrum manages overlay z-index automatically via its `OverlayProvider`. Ensure your application root includes `Provider` (or `ThemeProvider`, which wraps it) at the top level to avoid stacking context conflicts.

### Controlled vs. uncontrolled

`DialogTrigger` supports both patterns:

```tsx
// Uncontrolled â€” DialogTrigger manages open state internally
<DialogTrigger defaultOpen={false}>
    <Button>Open</Button>
    <Dialog>...</Dialog>
</DialogTrigger>;

// Controlled â€” you own the state
const [isOpen, setIsOpen] = useState(false);

<DialogTrigger isOpen={isOpen} onOpenChange={setIsOpen}>
    <Button onPress={() => setIsOpen(true)}>Open</Button>
    <Dialog>...</Dialog>
</DialogTrigger>;
```

Use the **controlled** pattern when you need to open or close the dialog programmatically (e.g., from a form validation result, a keyboard shortcut, or a parent component's state machine).

### The `close` callback

When `children` is a render function, `DialogTrigger` injects a `close()` callback. Calling it transitions the open state to `false` and triggers `onOpenChange(false)`. This is the recommended way to close a dialog from action buttons inside it:

```tsx
<DialogTrigger>
    <Button>Open</Button>
    {(close) => (
        <Dialog>
            <Content>Are you sure?</Content>
            <Footer>
                <ButtonGroup>
                    <Button variant="secondary" onPress={close}>
                        Cancel
                    </Button>
                    <Button
                        variant="negative"
                        onPress={() => {
                            handleDelete();
                            close();
                        }}
                    >
                        Delete
                    </Button>
                </ButtonGroup>
            </Footer>
        </Dialog>
    )}
</DialogTrigger>
```

## Related components

- **`AlertDialog`** â€” use for fixed-structure confirmation prompts (title + body text + primary/cancel buttons); use `Dialog` when you need custom slot composition, embedded forms, or a dismissible close button.
- **`DialogContainer`** â€” use when the trigger element may unmount before the dialog closes, or when the dialog is opened programmatically without a persistent trigger button.
- **`Popover`** â€” use for non-blocking, light-dismiss overlays anchored to a trigger; reserve `Dialog` for interactions that must block the rest of the UI.
- **`FullscreenAction`** â€” use when the dialog should occupy the full viewport; `FullscreenAction` composes the fullscreen trigger, tooltip, and dialog structure in one component.

## Example use cases

<details>
  <summary>Typical use cases</summary>

- **Create / rename form**: A "New project" or "Rename dataset" dialog contains a text field and Save/Cancel buttons, keeping the main view intact while the user enters a name.
- **Destructive confirmation with extra context**: Use when the confirmation message includes multiple paragraphs, a code block, or an embedded preview that `AlertDialog`'s plain text body cannot accommodate.
- **Detail view overlay**: Clicking a thumbnail opens a `Dialog size="L"` containing a full-resolution image viewer and annotation summary.
- **Multi-field settings form**: A `Dialog` with several form fields (dropdowns, sliders, checkboxes) for configuring export options or inference parameters.
- **Permission / terms acknowledgement**: An `isDismissable={false}` modal that the user must explicitly accept before proceeding.
