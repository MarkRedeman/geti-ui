# Plan: Migrate docs away from Storybook-coupled rendering

## Goal

Make `documentation/` the primary documentation + verification surface for `@geti/ui` by:

1. authoring docs pages directly in docs (no `*.stories` imports),
2. rendering explicit examples from `@geti/ui` components,
3. enforcing docs coverage in CI,
4. keeping Storybook temporarily for comparison/validation until cutover.

---

## Format decision: MDX (not TSX)

Use `documentation/docs/components/{category}/{Component}.mdx`.

Why:
- Rspress is MDX-first (TOC, search indexing, markdown UX, frontmatter).
- Our pages are prose + examples, not app screens.
- MDX still allows inline React examples and imported helper components.

When TSX is acceptable:
- only for extracted reusable demo helpers under `documentation/docs/components/_examples/**`.

---

## Target architecture

### Before
- Docs plugin discovers MDX in `packages/ui/src/components/**`.
- Stories are imported and rendered via `StoriesGallery`.
- Docs behavior is coupled to Storybook structure and story semantics.

### After
- Source of truth pages live in `documentation/docs/components/**`.
- Each page contains its own examples directly using `@geti/ui`.
- No imports from `*.stories.tsx` in docs runtime.
- Storybook remains optional dev sandbox, not docs dependency.

---

## Migration phases

## Phase 0 — Scaffolding and guardrails

1. Create directory structure for authored pages:
   - `documentation/docs/components/ui/**`
   - `documentation/docs/components/form/**`
   - `documentation/docs/components/form/pickers/**`
   - `documentation/docs/components/form/date-controls/**`
   - `documentation/docs/components/form/color-controls/**`
   - `documentation/docs/components/data/**`
   - `documentation/docs/components/feedback/**`
   - `documentation/docs/components/navigation/**`
   - `documentation/docs/components/layouts/**`
   - `documentation/docs/components/overlays/**`

2. Add shared docs helpers folder:
   - `documentation/docs/components/_examples/**` (optional extracted demo components).

3. Add page template in docs repo:
   - `documentation/docs/components/_template.mdx`
   - Uses section order from `plans/high-quality-user-documentation.md`.

4. Add migration tracking file:
   - `documentation/docs/components/_coverage.json` (generated by script).

Gate:
- docs build still passes,
- no behavior changes yet.

---

## Phase 1 — Dual-mode routing (safe transition)

Update `documentation/plugins/component-docs.ts` resolution order:

1. If docs-side page exists at `documentation/docs/components/{category}/{Component}.mdx`, use it.
2. Else fall back to current legacy pipeline (package MDX + stories injection).

This allows gradual migration without breaking existing pages.

Gate:
- existing routes stay valid,
- migrated pages are served from docs-side files.

Rollback:
- remove docs-side page(s); route falls back automatically.

---

## Phase 2 — Component page migration by waves

Migrate in this order:

1. **Wave A:** `ui/` + core `form/`
2. **Wave B:** `form/pickers`, `form/date-controls`, `form/color-controls`
3. **Wave C:** `data/` + `feedback/`
4. **Wave D:** `navigation/` + `layouts/` + `overlays/`

Per component page requirements:
- imports examples directly from `@geti/ui`,
- no `.stories` imports,
- includes template sections in expected order,
- examples are explicit and user-focused.

Gate per wave:
- docs build passes,
- no runtime errors across migrated routes,
- coverage check updated.

### Pilot notes from `ui/Button` migration

We migrated `Button` first as a reference implementation. These are the concrete
changes/patterns we should reuse for all next components:

1. **Authored page path**
   - New source page: `documentation/docs/components/ui/Button.mdx`
   - Legacy page remains untouched in `packages/ui/src/components/ui/Button/Button.mdx`
     for rollback until full cutover.

2. **Dual-mode route behavior worked as intended**
   - Route `/components/ui/Button` now resolves to authored docs page when present.
   - If authored page is removed, plugin falls back to legacy story-injected page.

3. **Example UX parity requirement**
   - Plain inline examples were not acceptable.
   - We introduced `documentation/theme/ExampleCard.tsx` to keep parity with old
     card-based example UX:
     - example title
     - live rendered example
     - show/hide code toggle

4. **Theme handling should be centralized**
   - `ThemeProvider` is applied inside `ExampleCard` (not repeated per example).
   - This keeps authored MDX concise and ensures consistent rendering/styling.

5. **Granularity of examples matters**
   - Variants/states should be separate cards (not grouped into one card) for
     better discoverability and cleaner code snippets.
   - For Button we split cards into: Accent, Primary, Secondary (outline),
     Negative, Pending, Disabled.

6. **Coverage tracking update per migrated component**
   - `documentation/docs/components/_coverage.json` must be updated when each
     component is migrated.
   - Current state after pilot: `Button` marked migrated (`1 / 83`).

7. **Commit strategy to keep history clear**
   - One commit for migration infra/scaffolding.
   - One commit per component migration.
   - Follow-up commits allowed for UX parity fixes discovered during review.

## Phase 3 — CI enforcement

Add CI checks:

1. **Docs coverage script** (`scripts/check-docs-coverage.ts`)
   - Enumerate components under `packages/ui/src/components/**`.
   - Assert docs page exists at `documentation/docs/components/**` for each component.
   - Exclude KitchenSink and helper-only files.

2. **No Storybook coupling script** (`scripts/check-docs-no-stories-imports.ts`)
   - Fail if any file under `documentation/**` imports `*.stories`.

3. **Docs build check**
   - `pnpm --filter @geti/documentation build`.

4. **Docs route smoke test**
   - Playwright script that visits all component routes and asserts no runtime crash.

5. **Dead-link check strictness**
   - remove Storybook-specific dead-link exclusions once legacy pages are gone.

---

## Phase 4 — Decommission Storybook coupling in docs

After 100% docs coverage:

1. Remove stories injection from `component-docs.ts`.
2. Remove `documentation/theme/StoriesGallery.tsx`.
3. Remove helper code that parses story source.
4. Keep sidebar generation (or replace with static config + route-check script).

At this point docs are independent from Storybook code.

---

## Phase 5 — Storybook role change (not immediate removal)

Storybook becomes optional developer sandbox for component development.

Keep:
- `*.stories.tsx` files,
- `.storybook` setup for local exploratory testing.

Remove from critical CI path only after stability period:
- Storybook build as required gate,
- any docs dependencies on Storybook output.

---

## Acceptance criteria

Migration is complete when all are true:

1. Every component has a docs page in `documentation/docs/components/**`.
2. No docs imports from `*.stories.tsx`.
3. `StoriesGallery` and story-injection plugin path are removed.
4. Docs build + smoke tests are green in CI.
5. Storybook is no longer required for docs correctness.

---

## Risks & mitigations

1. **Risk:** copy/paste low-quality examples during mass migration.
   - **Mitigation:** wave-based review and checklist enforcement.

2. **Risk:** route drift between sidebar and files.
   - **Mitigation:** route coverage script in CI.

3. **Risk:** hidden runtime failures in explicit examples.
   - **Mitigation:** Playwright smoke pass over all component routes.

4. **Risk:** long-running migration blocks progress.
   - **Mitigation:** dual-mode fallback allows incremental shipping.
