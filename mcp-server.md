# Plan: @geti-ai/mcp — Geti UI MCP Server

## Overview

Build an MCP (Model Context Protocol) server that exposes the Geti UI documentation to
AI coding agents. The server reads from the `llms.txt` and markdown files generated by
the `@rspress/plugin-llms` plugin during the documentation build, and serves them via
the MCP stdio transport protocol.

The implementation is modeled after Adobe's React Spectrum MCP server (`adobe/mcp/`)
but adapted for our single-library, local-first use case.

### Goals

- Let AI agents (Claude Code, Cursor, VS Code Copilot, Windsurf, etc.) discover and
  read Geti UI component docs, chart docs, and smart-tools docs via MCP tools
- Ship a self-contained npm package (`@geti-ai/mcp`) with all documentation bundled —
  `npx @geti-ai/mcp@latest` works out of the box with zero configuration
- Support local development via env var overrides for custom doc directories or remote
  URLs

## Architecture

```
                             ┌────────────────────────────┐
                             │  AI Agent                   │
                             │  (Claude, Cursor, VS Code)  │
                             └──────────┬─────────────────┘
                                        │ MCP (stdio JSON-RPC)
                                        ▼
                             ┌────────────────────────────┐
                             │   @geti-ai/mcp server       │
                             │                            │
                             │  7 tools:                  │
                             │   list_geti_ui_pages       │
                             │   get_geti_ui_page_info    │
                             │   get_geti_ui_page         │
                             │   search_geti_ui_docs      │
                             │   list_geti_ui_components  │
                             │   get_geti_ui_component_   │
                             │     props                  │
                             │   list_geti_ui_charts      │
                             └──────────┬─────────────────┘
                                        │
                     ┌──────────────────┼──────────────────┐
                     ▼                  ▼                  ▼
              Bundled data        Local dir             Remote URL
              (dist/data/)        (DOCS_DIR)           (DOCS_BASE_URL)
```

### Content resolution priority

1. **`DOCS_BASE_URL`** env var set → fetch all content over HTTP from a deployed docs
   site
2. **`DOCS_DIR`** env var set → read from that local directory (e.g. a custom
   `doc_build/` path)
3. **Default** → read from `dist/data/` bundled inside the npm package

This means the published npm package is fully self-contained — no local build or running
server required.

## Content source: `@rspress/plugin-llms`

The `@rspress/plugin-llms` (v2.0.4) is already configured in `documentation/rspress.config.ts`.
Running `rspress build` generates:

| File | Content |
|------|---------|
| `doc_build/llms.txt` | 193-line index: page titles, paths, descriptions structured by nav section |
| `doc_build/llms-full.txt` | 23,155-line concatenation of all page content |
| `doc_build/**/*.md` | 170 individual markdown files (one per doc page) |

The MCP server uses `llms.txt` as the page index and individual `.md` files for
per-page content. The `llms-full.txt` is not directly used at runtime but is bundled
for potential future use.

## MCP tools

### Core documentation tools (3)

These follow the same pattern as Adobe's React Spectrum MCP server.

| # | Tool | Input Schema | Description |
|---|------|-------------|-------------|
| 1 | `list_geti_ui_pages` | `{ includeDescription?: boolean }` | Lists all documentation pages from llms.txt. Returns sorted array of `{name, path}` or `{name, path, description}`. |
| 2 | `get_geti_ui_page_info` | `{ page_name: string }` | Returns page name, description, and list of `## ` section titles for a given page. |
| 3 | `get_geti_ui_page` | `{ page_name: string, section_name?: string }` | Returns full markdown content for a page, or a specific `## ` section if `section_name` is provided. |

### Search tool (1)

| # | Tool | Input Schema | Description |
|---|------|-------------|-------------|
| 4 | `search_geti_ui_docs` | `{ query: string, limit?: number }` | Case-insensitive text search across all page content. Returns matching pages with line number and snippet context. Default limit: 10. |

The search tool loads all markdown content on first invocation and caches it. Search is
substring-based (case-insensitive) across all pages, returning the most relevant matches
with surrounding context lines.

### Component-specific tools (2)

These parse structured data from the standardized component doc template used across all
83 UI component pages.

| # | Tool | Input Schema | Description |
|---|------|-------------|-------------|
| 5 | `list_geti_ui_components` | `{ category?: string }` | Lists all UI components with name, category, and one-line description. Categories: `ui`, `form`, `form/date-controls`, `form/color-controls`, `form/pickers`, `data`, `feedback`, `layouts`, `navigation`, `overlays`. |
| 6 | `get_geti_ui_component_props` | `{ component_name: string }` | Extracts structured prop information from a component page. Returns props grouped by category (Content, Value, Events, Accessibility, Validation) with descriptions. |

Component pages follow a consistent template with these extractable sections:
- **Import statement** — canonical import from `@geti-ai/ui`
- **Description** — first paragraph after `# Heading`
- **Responsibility** — what the Geti wrapper adds/changes vs upstream
- **Variants and states** — markdown tables (parseable into structured data)
- **Props (deep dive)** — bullet lists under categorized `### ` headings, format:
  `` * `propName`: description. ``
- **Related components** — bullet list of related components
- **Example use cases** — collapsible `<details>` sections

### Chart-specific tool (1)

| # | Tool | Input Schema | Description |
|---|------|-------------|-------------|
| 7 | `list_geti_ui_charts` | `{ type?: 'composition' \| 'primitive' }` | Lists all chart components with name, type (composition or primitive), and description. |

Chart pages use a different template with a formal props table
(`Prop | Type | Default | Description`) that can also be parsed by `get_geti_ui_page`
with `section_name: "Props"`.

## File-by-file implementation details

### `src/types.ts`

Core type definitions adapted from Adobe's `shared/src/types.ts`:

```ts
export type SectionInfo = {
  name: string;
  startLine: number;   // 0-based, inclusive
  endLine: number;     // 0-based, exclusive
};

export type PageInfo = {
  key: string;         // e.g. "components/ui/Button"
  name: string;        // display name from llms.txt or # heading
  description?: string;
  filePath: string;    // relative path to .md file (e.g. "components/ui/Button.md")
  sections: SectionInfo[];
};

export type ContentSource =
  | { type: 'bundled' }
  | { type: 'local'; dir: string }
  | { type: 'remote'; baseUrl: string };
```

### `src/utils.ts`

Utility functions:

- `errorToString(err: unknown): string` — safe error stringification (from Adobe)
- `resolveContentSource(): ContentSource` — checks env vars, returns content source config
- `resolveDataDir(): string` — for bundled mode, resolves `dist/data/` relative to the
  package installation

### `src/content-reader.ts`

Abstraction layer that reads content from any of the three sources:

```ts
export async function readContent(relativePath: string): Promise<string>;
```

- **Bundled**: `fs.readFile(path.join(dataDir, relativePath))`
- **Local**: `fs.readFile(path.join(localDir, relativePath))`
- **Remote**: `fetch(baseUrl + '/' + relativePath)` with 15s timeout

The content source is resolved once at startup and cached.

### `src/parser.ts`

Markdown parsing utilities (adapted from Adobe's `shared/src/parser.ts`):

- `parseSectionsFromMarkdown(lines: string[]): SectionInfo[]` — extracts `## ` headings
  as section boundaries, ignoring headings inside fenced code blocks
- `extractNameAndDescription(lines: string[]): { name: string; description?: string }` —
  extracts the first `# ` heading and the first non-empty paragraph after it
- `extractComponentProps(lines: string[]): Record<string, PropInfo[]>` — parses bullet
  lists under `## Props (deep dive) > ### Category` into structured prop objects
- `extractPropsTable(lines: string[]): PropInfo[]` — parses formal markdown tables
  (`Prop | Type | Default | Description`) used in chart docs

### `src/page-manager.ts`

Page index management (adapted from Adobe's `shared/src/page-manager.ts`):

- `buildPageIndex(): Promise<PageInfo[]>` — reads `llms.txt`, parses each
  `- [Name](path.md): Description` line, caches results. Idempotent.
- `resolvePageRef(pageName: string): Promise<PageInfo>` — resolves a user-provided page
  name (e.g. `"Button"`, `"components/ui/Button"`) to a cached `PageInfo`. Supports
  fuzzy matching: tries exact match, then path suffix match, then case-insensitive.
- `ensureParsedPage(info: PageInfo): Promise<PageInfo>` — lazily loads and parses page
  content to populate `sections` and `description`.

Unlike Adobe's implementation, we don't need a `library` parameter since we only have one
library. Page keys use the full relative path (e.g. `"components/ui/Button"`) instead of
a library-prefixed key.

### `src/search.ts`

Full-text search implementation:

- `searchDocs(query: string, limit?: number): Promise<SearchResult[]>` — loads all page
  content (cached after first load), performs case-insensitive substring search, returns
  matching pages with line numbers and context snippets.

```ts
type SearchResult = {
  pageName: string;
  pagePath: string;
  matches: Array<{
    lineNumber: number;
    lineContent: string;
  }>;
};
```

### `src/component-tools.ts`

Component-specific tool implementations:

- `listComponents(category?: string)` — filters page index to
  `components/{category}/**` paths, extracts name and description
- `getComponentProps(componentName: string)` — resolves component page, extracts props
  from the `## Props (deep dive)` section using `extractComponentProps()`

### `src/chart-tools.ts`

Chart-specific tool implementations:

- `listCharts(type?: 'composition' | 'primitive')` — filters page index to
  `charts/compositions/**` or `charts/primitives/**` paths

### `src/server.ts`

MCP server factory (adapted from Adobe's `shared/src/server.ts`):

```ts
export async function startServer(version: string): Promise<void> {
  const server = new McpServer({
    name: 'geti-ui-docs-server',
    version,
  });

  // Build page index at startup
  await buildPageIndex();

  // Register all 7 tools
  registerCoreTools(server);
  registerSearchTool(server);
  registerComponentTools(server);
  registerChartTools(server);

  // Connect stdio transport
  const transport = new StdioServerTransport();
  await server.connect(transport);
}
```

### `src/index.ts`

CLI entry point:

```ts
#!/usr/bin/env node
import { startServer } from './server.js';

if (process.argv.includes('--help') || process.argv.includes('-h')) {
  console.log(`@geti-ai/mcp - Geti UI documentation MCP server`);
  console.log(`\nUsage: npx @geti-ai/mcp`);
  console.log(`\nEnvironment variables:`);
  console.log(`  DOCS_BASE_URL  Fetch docs from a remote URL instead of bundled data`);
  console.log(`  DOCS_DIR       Read docs from a local directory instead of bundled data`);
  process.exit(0);
}

startServer('0.1.0');
```

### `scripts/bundle-docs.mjs`

Pre-publish script that copies documentation build output into the package distribution:

1. Resolve `documentation/doc_build/` relative to repo root
2. Copy `llms.txt` → `dist/data/llms.txt`
3. Copy `llms-full.txt` → `dist/data/llms-full.txt`
4. Recursively copy all `**/*.md` → `dist/data/**/*.md` (preserving directory structure)

This runs as part of the build pipeline so the published npm package contains all docs.

### `rslib.config.ts`

Key decisions:
- **ESM only** — the MCP SDK is ESM, and this is a CLI tool (no need for CJS)
- **Bundle: true** — produces a single file for fast `npx` startup
- **Target: node** — Node.js runtime, not browser
- **Shebang banner** — makes the output executable as a CLI
- **External deps** — the MCP SDK is externalized (peer dependency pattern)

### `package.json`

- **`engines.node >= 18`** (not 24 like the monorepo root) since consumers install this
  package externally and may use older Node.js versions. The MCP SDK requires >= 18.
- The `bin` field enables `npx @geti-ai/mcp` usage.
- The `build` script runs `bundle-docs.mjs` first, then `rslib build`.

## CI integration

### Build & test (ci.yml)

Add MCP type-check and build steps to the existing CI workflow:

```yaml
# In jobs.ci.steps:
- name: Type check (mcp)
  run: npm run type-check
  working-directory: packages/mcp

- name: Build MCP server
  run: npm run build
  working-directory: packages/mcp
```

The MCP build depends on `docs:build` having been run (to generate `doc_build/`). In CI,
this is handled by the existing `docs` job. The MCP build step should either:
- Run after the `docs` job (use `needs: [docs]`)
- Or include its own `rspress build` step in the build script

Recommended approach: add a dedicated step that builds docs first, then the MCP package.

### Release workflow (release-mcp.yml)

Create a new release workflow following the same pattern as `release-ui.yaml`,
`release-charts.yml`, and `release-smart-tools.yml`:

```yaml
name: Release mcp

on:
  push:
    branches: [main]

concurrency:
  group: release-mcp-${{ github.ref }}
  cancel-in-progress: false

permissions: {}

jobs:
  release:
    name: Publish @geti-ai/mcp
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write

    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
          persist-credentials: false

      - uses: actions/setup-node@v6
        with:
          node-version: 24
          registry-url: https://registry.npmjs.org
          package-manager-cache: false

      - run: npm ci

      # Build all library packages (needed for docs build)
      - run: npm run build

      # Build documentation (generates llms.txt + markdown files)
      - run: npm run docs:build

      # Build MCP server (bundles docs into dist/data/, then compiles)
      - run: npm run build
        working-directory: packages/mcp

      - uses: changesets/action@v1
        id: changesets
        with:
          publish: npm publish --workspace=@geti-ai/mcp
          version: npx changeset version
          commit: "chore(release): version mcp"
          title: "chore(release): version mcp"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

      - name: Generate release notes with git-cliff
        if: steps.changesets.outputs.published == 'true'
        uses: orhun/git-cliff-action@v4
        with:
          config: cliff.toml
          args: --latest --strip header --tag-pattern "mcp-v[0-9].*"
        env:
          OUTPUT: RELEASE_NOTES.md
          GITHUB_REPO: ${{ github.repository }}

      - name: Resolve published version
        if: steps.changesets.outputs.published == 'true'
        id: release_meta
        uses: actions/github-script@v8
        env:
          PUBLISHED_PACKAGES: ${{ steps.changesets.outputs.publishedPackages }}
        with:
          script: |
            const packages = JSON.parse(process.env.PUBLISHED_PACKAGES || '[]');
            const target = packages.find((p) => p.name === '@geti-ai/mcp') ?? packages[0];
            if (!target?.version) {
              core.warning('Package not published; skipping GitHub Release.');
              core.setOutput('version', '');
              return;
            }
            core.setOutput('version', target.version);

      - name: Create GitHub Release
        if: >-
          steps.changesets.outputs.published == 'true'
          && steps.release_meta.outputs.version != ''
          && hashFiles('RELEASE_NOTES.md') != ''
        uses: softprops/action-gh-release@v2
        with:
          body_path: RELEASE_NOTES.md
          tag_name: ${{ format('mcp-v{0}', steps.release_meta.outputs.version) }}
          token: ${{ secrets.GITHUB_TOKEN }}
```

### Keeping docs content fresh in releases

Since the MCP package bundles documentation, it should be re-released whenever the
documentation content changes meaningfully. Two strategies:

1. **Changesets linked versioning (recommended):** Update the changeset config's
   `"linked"` array so that when any package in the group gets a version bump, the MCP
   package also gets bumped:

   ```json
   {
     "linked": [
       ["@geti-ai/ui", "@geti-ai/charts", "@geti-ai/smart-tools", "@geti-ai/mcp"]
     ]
   }
   ```

2. **Workflow dispatch:** Add a `workflow_dispatch` trigger to `release-mcp.yml` so the
   MCP package can be manually re-released after docs-only changes.

## Differences from Adobe's implementation

| Aspect | Adobe MCP | Geti MCP |
|--------|-----------|----------|
| **Libraries** | Two servers (s2, react-aria) with shared code | Single server for all Geti packages |
| **Content source** | Always HTTP from CDN | Local-first: bundled files, optional local dir or remote URL |
| **Build tool** | Plain `tsc` | rslib (repo convention) |
| **Package structure** | 3 packages (shared, s2, react-aria) | 1 self-contained package |
| **Tools** | 3 core + 3 S2-specific | 3 core + 1 search + 2 component + 1 chart |
| **Bundled data** | S2 bundles icon/illustration JSON | Bundles entire docs (llms.txt + all .md files) |
| **Transport** | stdio | stdio (same) |
| **Library prefix** | `s2` / `react_aria` | `geti_ui` |

## Future improvements

### Short-term (post-initial release)

- **Smoke test script** — Port Adobe's `smoke-list-pages.mjs` pattern: a small MCP
  client that spawns the server, calls each tool, and verifies responses. Run in CI.
- **`get_geti_ui_chart_props`** — Dedicated tool that parses the formal
  `Prop | Type | Default | Description` table from chart docs and returns structured
  JSON. Currently achievable via `get_geti_ui_page` with `section_name: "Props"` but a
  dedicated tool would return parsed data instead of raw markdown.
- **`get_geti_ui_component_examples`** — Extract `tsx preview` code blocks from a
  component page and return them as named examples. Useful for agents that want to
  generate code following existing patterns.

### Medium-term

- **MCP Resources** — In addition to tools, register MCP
  [resources](https://modelcontextprotocol.io/docs/concepts/resources) for
  `llms.txt` and `llms-full.txt` so agents can subscribe to them directly.
- **MCP Prompts** — Register MCP
  [prompts](https://modelcontextprotocol.io/docs/concepts/prompts) like
  `use_geti_ui_component` that provides a structured prompt template with component
  guidelines, import patterns, and theming rules.
- **Automated `llms.txt` validation** — CI step that builds docs, starts the MCP server,
  and verifies all pages in `llms.txt` are resolvable and parseable.
- **Component search by category/tag** — Extend `list_geti_ui_components` to support
  filtering by tags (e.g. `"accessible"`, `"spectrum-wrapped"`, `"geti-native"`).

### Long-term

- **Streamable HTTP transport** — In addition to stdio, support the MCP
  [Streamable HTTP](https://modelcontextprotocol.io/docs/concepts/transports#streamable-http)
  transport so the server can be deployed as a service (e.g. for web-based AI tools).
- **Playwright MCP integration** — Connect to a running Storybook instance via
  [Playwright MCP](https://github.com/microsoft/playwright-mcp) to enable visual testing
  and screenshot-based component verification from AI agents.
- **Live type extraction** — Instead of parsing markdown props, extract prop types
  directly from TypeScript source using the TS compiler API (similar to Adobe's
  `build-data.mjs` approach for style macros). This would produce accurate, always-fresh
  prop definitions.
- **Automatic re-publish on docs deploy** — Trigger MCP package release from the
  `deploy-main.yml` workflow after docs are deployed to GitHub Pages, ensuring the
  bundled content always matches the live documentation site.
- **AgentSkill integration** — Connect the MCP server to the planned
  `.agents/skills/geti-ui/` skill definition (referenced in `AGENTS.md`) so that agent
  frameworks that support skills can automatically discover and use the MCP server.
