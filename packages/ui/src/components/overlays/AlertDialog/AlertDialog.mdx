# AlertDialog

AlertDialogs are special types of dialogs that present a prompt to the user to confirm a specific action.

```tsx
import { AlertDialog, DialogTrigger } from '@geti/ui';
```

- [React Spectrum V3 AlertDialog](https://react-spectrum.adobe.com/react-spectrum/AlertDialog.html)

<Stories />

## Responsibility

`AlertDialog` is a **specialized confirmation dialog** — a thin wrapper around Spectrum's `AlertDialog` that enforces a standardized layout with a title, a body, and a fixed set of action buttons (Primary, Secondary, and Cancel). Unlike the freeform `Dialog`, `AlertDialog` has no slots: its structure is predetermined and its buttons are declarative props.

It is always used as a direct child of `DialogTrigger`, which owns the open state and manages the overlay lifecycle:

```tsx
<DialogTrigger>
    <Button variant="negative">Delete project</Button>
    <AlertDialog
        variant="destructive"
        title="Delete project?"
        primaryActionLabel="Delete"
        cancelLabel="Cancel"
        onPrimaryAction={() => handleDelete()}
    >
        This will permanently delete the project and all associated data. This action cannot be undone.
    </AlertDialog>
</DialogTrigger>
```

Key responsibilities:
- **Critical Feedback**: Provides clear visual and semantic cues for the severity of the action (e.g., a red primary button for the `"destructive"` variant).
- **Standardized Actions**: Ensures consistent button placement and labeling across the application without requiring manual `Footer` / `ButtonGroup` composition.
- **Accessibility**: Automatically assigns `role="alertdialog"` and manages focus transitions, including `autoFocusButton`.

### When to use
- For **destructive, irreversible actions** that require explicit user confirmation — e.g., "Delete project", "Overwrite file", "Revoke access".
- For **critical warnings** where the user must acknowledge a risk before proceeding — e.g., "Proceeding will invalidate your current session".
- When the confirmation content is **simple body text** with no embedded forms, tables, or rich UI elements.
- When you need a clear "Yes / No" or "Confirm / Cancel" pattern and want Spectrum's built-in button styling (red for destructive, etc.) automatically applied.

### When to avoid
- For **complex forms or rich content** — use `Dialog` with full slot composition instead, which gives you control over the interior structure.
- For **non-critical acknowledgements** — use `Toast` for transient feedback, or `Dialog` with an `"information"` intent when a less urgent overlay is sufficient.
- When you need a **dismissible close button** (×) in the corner — `AlertDialog` does not support this; use `Dialog` with `isDismissable` on `DialogTrigger`.
- When the action is **low-stakes and reversible** — avoid interrupting the user flow with a modal; prefer inline affordances.

## Design

`AlertDialog` follows the same *transparent wrapper* pattern as `Dialog`: it extends `SpectrumAlertDialogProps` without modification and delegates all rendering to Spectrum.

```tsx
// packages/ui/src/components/overlays/AlertDialog/AlertDialog.tsx

export interface AlertDialogProps extends SpectrumAlertDialogProps {}

export const AlertDialog = (props: AlertDialogProps) => <SpectrumAlertDialog {...props} />;
```

All props are passed directly through to `SpectrumAlertDialog` via spread. This keeps the Geti wrapper as a stable integration point — it can later adopt custom behaviour (e.g., analytics callbacks, migration to `react-aria-components`) without changing the consumer API.

`AlertDialog` is always placed as the second child of `DialogTrigger`. `DialogTrigger` renders it inside a modal overlay, controls its open/close state, and wires up focus trapping and dismissal. `AlertDialog` itself carries no open/closed concept — it only defines what is displayed and which action handlers to call.

**Prop-passing pattern**

Since `AlertDialogProps` is a direct extension of `SpectrumAlertDialogProps`, every prop documented in the [Spectrum AlertDialog API](https://react-spectrum.adobe.com/react-spectrum/AlertDialog.html) is available. The most commonly used are declared as explicit props in the table below, but any undocumented Spectrum prop can be passed through because the type definition (`extends SpectrumAlertDialogProps`) inherits the full upstream surface.

## Variants

### Confirmation (default)
The standard variant for reversible or low-risk confirmations. The primary button uses the default accent color.

```tsx
<AlertDialog
    variant="confirmation"
    title="Save changes?"
    primaryActionLabel="Save"
    cancelLabel="Cancel"
>
    Your changes will be saved permanently.
</AlertDialog>
```

### Destructive
Used for actions that result in data loss or are irreversible. The primary action button is automatically styled with a negative (red) color to signal danger.

```tsx
<AlertDialog
    variant="destructive"
    title="Delete dataset?"
    primaryActionLabel="Delete"
    cancelLabel="Cancel"
>
    This will permanently delete the dataset and all of its annotations.
</AlertDialog>
```

### Information
Presents a non-actionable notice with a single "Got it" button. Use when you need to inform the user of something important before they continue.

### Warning
Includes a warning icon in the heading. Use when the user is about to do something that may have unintended side effects but is not necessarily destructive.

### Error
Communicates that something has gone wrong. Typically triggered programmatically rather than by a user action.

### With Secondary Action
Adds a third button between the primary action and Cancel. Use when there is a meaningful alternative action alongside the primary one (e.g., "Save" / "Don't save" / "Cancel").

```tsx
<AlertDialog
    variant="confirmation"
    title="Save document?"
    primaryActionLabel="Save"
    secondaryActionLabel="Don't save"
    cancelLabel="Cancel"
    onPrimaryAction={() => save()}
    onSecondaryAction={() => discard()}
    onCancel={() => {/* stay in the dialog's caller context */}}
>
    Do you want to save your changes before closing?
</AlertDialog>
```

## Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `title` | `string` | — | The title of the alert dialog, rendered in the dialog header and used as the accessible label. |
| `children` | `ReactNode` | — | The message or body text of the alert dialog. Keep this concise — for complex content use `Dialog`. |
| `variant` | `'confirmation' \| 'information' \| 'destructive' \| 'error' \| 'warning'` | `'confirmation'` | Controls the visual style and semantic intent. `'destructive'` renders the primary button in red. |
| `primaryActionLabel` | `string` | — | The label for the primary action button. Always required. |
| `secondaryActionLabel` | `string` | — | The label for an optional secondary action button, rendered between the primary and cancel buttons. |
| `cancelLabel` | `string` | — | The label for the cancel button. If omitted, no cancel button is rendered. |
| `onPrimaryAction` | `() => void` | — | Handler called when the primary action button is pressed. The dialog closes automatically after this returns. |
| `onSecondaryAction` | `() => void` | — | Handler called when the secondary action button is pressed. The dialog closes automatically after this returns. |
| `onCancel` | `() => void` | — | Handler called when the cancel button is pressed or Escape is pressed. The dialog closes automatically after this returns. |
| `autoFocusButton` | `'primary' \| 'secondary' \| 'cancel'` | — | Which button receives focus when the dialog opens. If unset, Spectrum determines focus from DOM order. |

## Accessibility

**Role**

Spectrum assigns `role="alertdialog"` to the dialog element. This role is specifically designed for dialogs that contain an alert, an error, or a prompt requiring an immediate response. Unlike `role="dialog"`, screen readers will typically interrupt ongoing announcements to read the alert dialog's title when it opens.

**Labeling**

The `title` prop is automatically wired as the accessible label for the dialog via `aria-labelledby`. Always provide a clear, action-oriented title that describes what is being confirmed (e.g., "Delete dataset?" rather than "Warning").

**Focus management**

When the dialog opens, Spectrum moves focus into it automatically. By default, focus lands on the first focusable element in DOM order (typically the primary action button). Use `autoFocusButton` to override this when a safer default is needed:

```tsx
{/* For destructive dialogs, focus "Cancel" by default to prevent accidental confirmation */}
<AlertDialog
    variant="destructive"
    title="Delete all annotations?"
    primaryActionLabel="Delete all"
    cancelLabel="Cancel"
    autoFocusButton="cancel"
>
    This will remove all annotations from the dataset.
</AlertDialog>
```

When the dialog closes, focus automatically returns to the trigger element — no manual focus restoration is required.

**Keyboard interactions**

| Key | Behaviour |
|-----|-----------|
| `Tab` | Move focus to the next button within the dialog |
| `Shift + Tab` | Move focus to the previous button |
| `Enter` / `Space` | Activate the focused button |
| `Escape` | Close the dialog and call `onCancel` |

**Screen reader behaviour**

When the dialog opens, screen readers interrupt the current reading flow and announce the dialog title and role. The `role="alertdialog"` landmark ensures users can navigate to it directly in their reader's dialog or landmark navigation. Content behind the dialog is marked as `aria-hidden` while the overlay is active.

## Integration

- **Dependencies**: `@adobe/react-spectrum` — `AlertDialog` and `SpectrumAlertDialogProps` are imported from this package. There is no custom rendering logic in `@geti/ui`'s wrapper.
- **DialogTrigger**: `AlertDialog` must be a direct child of `DialogTrigger` to function correctly. `DialogTrigger` is exported from `@geti/ui` alongside `AlertDialog`. Do not attempt to render `AlertDialog` standalone — it requires the Spectrum overlay context provided by `DialogTrigger`.
- **vs. Dialog**: Use `AlertDialog` when you need a fixed-structure confirmation with Spectrum's built-in button styling. Use `Dialog` when you need custom slot composition, embedded forms, or a dismissible close button. The two components are mutually exclusive — do not nest one inside the other.
- **ThemeProvider**: Must be present in the React tree. The overlay background, border radius, shadow, and button colors all derive from Spectrum tokens resolved by `ThemeProvider`.
- **Migration note**: Projects that previously used custom Intel-branded confirmation dialogs (non-Spectrum modals with manual button composition) should migrate to `AlertDialog` + `DialogTrigger` for critical confirmations. This ensures consistent Spectrum token usage, correct `role="alertdialog"` semantics, and automatic focus management without custom implementation.

### Automatic close behaviour

When `AlertDialog` is used inside `DialogTrigger`, pressing any action button (primary, secondary, or cancel) automatically closes the dialog. You do **not** need to call a `close()` function manually. Your action handlers receive the button press event and can perform side effects freely; the overlay is dismissed by `DialogTrigger` after the handler returns.

```tsx
<DialogTrigger>
    <Button variant="negative">Delete model</Button>
    <AlertDialog
        variant="destructive"
        title="Delete model?"
        primaryActionLabel="Delete"
        cancelLabel="Cancel"
        onPrimaryAction={() => {
            // No close() call needed — DialogTrigger handles it automatically.
            deleteModel(modelId);
        }}
        onCancel={() => {
            // Called on cancel button press or Escape key.
            // The dialog will close after this handler returns.
        }}
    >
        Deleting this model cannot be undone.
    </AlertDialog>
</DialogTrigger>
```

This contrasts with `Dialog`, where you must compose your own `ButtonGroup` and explicitly call the injected `close()` callback in each button's `onPress` handler.

### Controlled open state

If you need to open the `AlertDialog` programmatically (e.g., from a validation result or keyboard shortcut), use the controlled pattern on `DialogTrigger`:

```tsx
const [isOpen, setIsOpen] = useState(false);

<DialogTrigger isOpen={isOpen} onOpenChange={setIsOpen}>
    <Button onPress={() => setIsOpen(true)}>Delete</Button>
    <AlertDialog
        variant="destructive"
        title="Delete item?"
        primaryActionLabel="Delete"
        cancelLabel="Cancel"
        onPrimaryAction={() => deleteItem()}
    >
        This action cannot be undone.
    </AlertDialog>
</DialogTrigger>
```

## Related components

- **`Dialog`** — use when the confirmation content requires custom slot composition, embedded forms, a dismissible close button, or content too rich for `AlertDialog`'s plain text body.
- **`DialogContainer`** — use `AlertDialog` as the child of `DialogContainer` when the trigger element may unmount before the dialog closes or when the dialog is opened programmatically.
- **`DialogTrigger`** — always the direct parent of `AlertDialog`; manages the open state and overlay lifecycle so `AlertDialog` doesn't have to.
- **`Tooltip`** — use alongside the destructive trigger button to hint at consequences before the user even opens the dialog.

## Example use cases

<details>
  <summary>Typical use cases</summary>

- **Delete confirmation**: A "Delete project" button opens a `variant="destructive"` `AlertDialog`; `autoFocusButton="cancel"` prevents accidental confirmation by focusing Cancel.
- **Overwrite warning**: Before importing a dataset that would replace an existing one, a `variant="warning"` dialog warns the user and offers "Overwrite" / "Cancel".
- **Unsaved changes prompt**: When navigating away from an edited page, a `variant="confirmation"` dialog with "Save", "Don't save", and "Cancel" uses the `secondaryActionLabel` slot.
- **Session expiry notice**: A `variant="information"` dialog with a single "OK" button informs the user their session is about to expire.
- **Bulk action confirmation**: Selecting multiple items and pressing "Delete selected" opens an `AlertDialog` listing the count of items to be removed.

</details>
