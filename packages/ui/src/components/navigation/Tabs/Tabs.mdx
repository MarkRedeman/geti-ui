## Responsibility

`Tabs` are used to organize **same-level sibling content** where only one section is visible at a time. They provide a clear visual hierarchy and allow users to switch between related content sets without leaving the current context.

### When to use
- **Related Content**: Grouping related information that forms a coherent unit (e.g., project settings, user profile tabs).
- **Single Visibility**: When only one panel needs to be seen at once to reduce cognitive load.
- **Limited Sections**: Best for 2–7 sections; for larger sets, consider a vertical navigation sidebar.

### Comparison with other components
| Situation | Use instead |
| :--- | :--- |
| Persistent navigation across route changes | `VerticalNav` |
| Items are actions that perform operations | `ActionGroup` |
| Large number of sections (> 8) | `VerticalNav` |
| Toggleable filters or multi-select states | `ToggleButtonGroup` |

### Usage Rules
- **Labeling**: Always provide an `aria-label` on the `TabList` (or `aria-labelledby` pointing to a heading).
- **Nesting**: Avoid nesting `Tabs` within `Tabs` as it complicates keyboard navigation and focus flow.
- **Orientation**: Default is horizontal; use `orientation="vertical"` for sidebar-style tabs.

## Variants

### Density
Supports `regular` (default) and `compact` density for tighter layouts.

### Orientation
Can be rendered horizontally or vertically. Vertical tabs are useful when vertical space is abundant but horizontal space is constrained.

### Tabs Props
| Prop | Type | Default | Description |
| :--- | :--- | :--- | :--- |

## Accessibility

### Roles and Navigation
- `Tabs` follows the WAI-ARIA Tabs pattern: `role="tablist"` for the container, `role="tab"` for each item, and `role="tabpanel"` for the content.
- **Keyboard Navigation**:
    - `Tab`: Enters the tab list (focuses the active tab). Subsequent `Tab` presses move focus **into the active panel**.
    - `Arrow Left / Right`: Moves focus and (by default) selects the next/previous tab.
    - `Home / End`: Jump to the first or last tab.
- **Activation Mode**: By default, moving focus with arrows selects the tab (`keyboardActivation="automatic"`). For expensive panel renders, use `keyboardActivation="manual"` to require `Enter` or `Space` to confirm.

### Labelling Requirement
Always include an `aria-label` or `aria-labelledby` on the `TabList` component to provide context to screen reader users.

## Spectrum 2 Migration

The Spectrum 2 (S2) migration involves significant structural changes to the JSX hierarchy:

- **Remove `<TabPanels>` wrapper**: In S2, individual `<TabPanel>` elements are siblings to the `<TabList>`, and the wrapper is eliminated.
- **Item Rename**: `<Item>` in the list becomes `<Tab>`, and in the panels, it becomes `<TabPanel>`.
- **Dynamic Collections**: The `items` prop moves from the root `<Tabs>` component to the `<TabList>`.
- **Automatic Overflow**: S2 automatically collapses overflowing tabs into a `Picker` dropdown menu.
- **Style macro**: V3 dimension props (e.g., `margin`, `width`) are replaced by the `styles={style({...})}` macro.

### S2 Migration Mapping
| V3 Component | S2 Equivalent |
| :--- | :--- |
| `<Tabs>` | `<Tabs>` |
| `<TabList>` | `<TabList>` |
| `<TabPanels>` | ❌ Removed |
| `<Item>` (in list) | `<Tab id="...">` |
| `<Item>` (in panels) | `<TabPanel id="...">` |
