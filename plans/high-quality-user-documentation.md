# Plan: High-Quality User Documentation Refactor

This plan defines how we will evolve Geti UI docs from implementation-centric pages into clear, task-oriented docs for product engineers.

## 🎯 Goal

For each component page, provide:

- clear decision guidance (when to use / not use)
- practical, copy-paste-able examples
- concise prop guidance (not API dump noise)
- links to canonical upstream docs where relevant

The target style should be closer to React Spectrum usage docs than autogenerated API references.

---

## ✅ Current baseline (as of now)

- Rspress docs app exists at `examples/documentation/`
- Component pages are sourced from component MDX files in `packages/ui/src/components/**`
- Story examples are rendered inline in docs via `StoriesGallery`
- Show code now renders per-story snippets (not full stories file)
- New guide pages exist under Components:
  - `/components/installation`
  - `/components/theming`

---

## 🧱 Page template specification

Each component MDX should follow this structure in order:

1. **Title + short description**
   - 1–2 sentences: what it is, and primary use case.

2. **Import block**
   - Canonical import from `@geti/ui`.

3. **Reference links**
   - Link to relevant React Spectrum / React Aria docs when applicable.

4. **Examples**
   - Rendered stories section (already injected by docs plugin).

5. **Responsibility / behavior notes**
   - What this wrapper changes vs upstream.

6. **Variants and states**
   - Practical differences, recommended defaults.

7. **Props (deep dive)**
   - Group by concern (content, value, events, validation, accessibility).
   - Focus on meaningful props and edge cases.

8. **Related components**
   - Mention similar/adjacent components and when to choose each.

9. **Example use cases (AI/accessibility-friendly section)**
   - Hidden/collapsible content is acceptable.
   - Include concrete usage contexts (pages, workflows, composite UI blocks).

---

## ❓Clarifications / ambiguity resolved

1. **"Exact structure" means section order, not rigid headings.**
   - Section titles can vary slightly, but content intent and ordering should match the template.

2. **Not every component needs a "Variants" section if variants do not exist.**
   - In that case, replace with "States" or omit with a short explanation.

3. **Props section should avoid duplicating full upstream API docs.**
   - Link upstream for exhaustive details; document only wrapper-specific or high-impact props in depth.

4. **Reference links are required when the component wraps Spectrum/Aria primitives.**
   - Optional only for purely Geti-native components.

5. **KitchenSink pages are not part of the component checklist.**
   - They are development artifacts and should not drive user-facing documentation scope.

6. **"Example use cases" should be concise.**
   - 3–8 bullets max, concrete and product-relevant; avoid vague filler.

## 🚀 Execution strategy

1. **Start with reference pages**
   - Keep `Installation` and `Theming` pages up to date.

2. **Iterate component-by-component**
   - Update one component doc page at a time.
   - Build/preview, then commit.

3. **Track reviews**
   - Record decisions and follow-up notes in `plans/documentation-review.md`.

4. **Phased rollout**
   - Phase 1: UI + core Form controls
   - Phase 2: Pickers + Date + Color controls
   - Phase 3: Data + Feedback
   - Phase 4: Navigation + Layouts + Overlays
   - Final audit: consistency, stale links, and doc quality pass
