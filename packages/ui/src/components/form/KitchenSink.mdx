## Responsibility

The `form/` category owns **all user-input primitives** — every component a user interacts with to enter or select data. This spans simple text fields through to a full multi-surface colour picker dialog. It is the primary surface for data capture, validation feedback, and file ingestion in the Geti design system.

Components in this folder:

| Component              | Purpose                                                                         |
| ---------------------- | ------------------------------------------------------------------------------- |
| `Form`                 | Form container — propagates `isDisabled` and `validationBehavior` to children   |
| `TextField`            | Single-line text input                                                          |
| `TextArea`             | Multi-line text input                                                           |
| `PasswordField`        | Text input with show/hide toggle, optional error message, and new-password hint |
| `SearchField`          | Search input with clear button and optional ref forwarding                      |
| `NumberField`          | Numeric input with stepper buttons                                              |
| `Checkbox`             | Single checkbox                                                                 |
| `CheckboxGroup`        | Group of related checkboxes                                                     |
| `RadioGroup` / `Radio` | Mutually exclusive radio button group                                           |
| `Switch`               | On/off toggle switch                                                            |
| `Slider`               | Single-handle range slider (Geti energy-blue fill override when `isFilled`)     |
| `RangeSlider`          | Dual-handle range slider                                                        |
| `SelectBoxGroup`       | Custom visual card-style selector group                                         |
| `DropZone`             | Drag-and-drop file target area                                                  |
| `FileTrigger`          | Attaches native file-picker to a child button                                   |
| **pickers/**           | Date, time, calendar, colour picker sub-components                              |

Sub-folder `pickers/` contains:

| Picker                              | Purpose                                                 |

| `ColorArea`                         | 2-D colour gradient control                             |
| `ColorSlider`                       | Single-channel colour slider                            |
| `ColorWheel`                        | Hue wheel picker                                        |
| `ColorField`                        | Hex/CSS colour text input                               |
| `ColorSwatch` / `ColorSwatchPicker` | Preset colour chips                                     |
| `ColorThumb`                        | Draggable colour handle (used in ColorArea/ColorSlider) |
| `ColorPickerDialog`                 | Composite modal combining all colour pickers            |

## Design

### Thin Spectrum wrapper (default pattern)

Most form inputs are thin wrappers over `@adobe/react-spectrum`:

```tsx
export interface TextFieldProps extends SpectrumTextFieldProps {}
export const TextField = (props: TextFieldProps) => <SpectrumTextField {...props} />;
```

No props are altered; the sole purpose is to centralise the import so consumers never reach into `@adobe/react-spectrum` directly. Type identity is preserved by extending the Spectrum interface directly.

### Behavioural extensions on Spectrum base

`PasswordField` is the most extended form component. It wraps `SpectrumTextField` but adds its own outer `<div>` to position a `<button>` show/hide toggle, manages `useState` for visibility, and computes `aria-describedby` from optional `error` and `isNewPassword` props. Custom styling lives in `PasswordField.module.css`.

`Slider` applies a Geti-specific CSS module class (`styles.slider`) always, plus a `data-filled` attribute toggled by the `isFilled` prop. The CSS module uses the `[data-filled="true"]` selector to apply `--energy-blue` override tokens to the fill track and thumb.

`SearchField` uses `forwardRef` to expose `TextFieldRef` for callers that need to programmatically focus/clear the field.

### FileTrigger sources from react-aria-components

`FileTrigger` is one of only two form components **not** wrapping `@adobe/react-spectrum`. It wraps `react-aria-components`' `FileTrigger` instead, because that primitive is only available in the RAC package.

### ColorPickerDialog — composite Geti component

`ColorPickerDialog` is a fully stateful composite that assembles `ColorArea`, `ColorSlider`, `ColorWheel`, `ColorField`, and `ColorSwatchPicker` from Spectrum into a `DialogTrigger → Dialog` pattern. It owns internal state (`useState<Color>`) for in-flight editing, calls `onColorChange` only on Confirm, and syncs back from `colorProp` via `useEffect`. This is the one form component with substantial business logic.

### Props pattern

- All wrapper components export their own `XxxProps` interface extending the Spectrum equivalent.
- `Omit` is used when a prop is intentionally excluded (`PasswordField` omits `type`, `SearchField` omits `ref`).
- Generic type parameters are forwarded where needed (`Picker<T>`, `ComboBox<T>`).

## Flow

1. **Uncontrolled / Spectrum-managed**: most inputs are uncontrolled by default (Spectrum manages their ARIA state internally). Consumers can make them controlled via `value` / `onChange`.
2. **`Form` propagates context**: wrapping inputs in `Form` allows `isDisabled` and `validationBehavior` to cascade down without per-component threading.
3. **`ColorPickerDialog` is controlled with internal draft state**: the colour shown in the dialog diverges from the confirmed value until the user presses Confirm.

---

## Integration

- **Depends on**: `@adobe/react-spectrum` (most inputs), `react-aria-components` (`FileTrigger`), `../ui/Button` (internal composition in `ColorPickerDialog`).
- **Consumed by**: application-level forms, filter panels, settings dialogs throughout Geti.
- **Overlays dependency**: `ColorPickerDialog` uses `Dialog`/`DialogTrigger` from Spectrum directly (not from `overlays/`), so it is self-contained.
- **Theme**: all inputs adopt Geti's dark-mode tokens automatically via `ThemeProvider`.
