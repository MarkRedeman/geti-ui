# Theming

<p>How Geti UI overrides Adobe React Spectrum's design token system to produce a dark-first, Intel Geti brand palette — and how components consume those tokens at every layer.</p>

---

## Overview

Geti UI is built on **Adobe React Spectrum v3**, which exposes a CSS custom property (CSS variable) based token system. Rather than forking Spectrum or reimplementing its visual layer, Geti **extends** the default theme by merging an additional set of CSS module files on top of Spectrum's own. The result is a single `getiTheme` object consumed by `ThemeProvider`, which is the only configuration point a consuming application needs.

```
ThemeProvider
  └── Spectrum Provider (getiTheme, colorScheme="dark", scale="medium")
        ├── geti-global.module.css  ──►  .spectrum        (always active)
        ├── geti-dark.module.css    ──►  .spectrum--dark   (colorScheme="dark")
        ├── geti-light.module.css   ──►  .spectrum--light  (colorScheme="light", stub)
        ├── geti-medium.module.css  ──►  .spectrum--medium (scale="medium", stub)
        └── geti-large.module.css   ──►  .spectrum--large  (scale="large", stub)
```

---

## File map

| File | Spectrum class | Purpose |
|---|---|---|
| `theme.ts` | — | Assembles `getiTheme` by merging Spectrum's `defaultTheme` with Geti layers via `mergeClasses` |
| `ThemeProvider.tsx` | — | React wrapper around Spectrum's `Provider`; sets `colorScheme="dark"`, `scale="medium"`, `locale="en-US"` |
| `geti-global.module.css` | `.spectrum` | Always-active tokens: brand palette, semantic overrides, animation durations, `@keyframes shimmer`, component-level typography tweaks |
| `geti-dark.module.css` | `.spectrum--dark` | Dark mode: Geti gray ramp, blue ramp, overlay/shadow aliases, toast palette, component colour overrides |
| `geti-light.module.css` | `.spectrum--light` | Light mode stub — currently empty; Geti is dark-first |
| `geti-medium.module.css` | `.spectrum--medium` | Medium-scale stub |
| `geti-large.module.css` | `.spectrum--large` | Large-scale stub |

## ThemeProvider

`ThemeProvider` is the **single required wrapper** for all Geti UI components. It provides the Geti theme tokens via React context.

```tsx
import { ThemeProvider } from '@geti/ui';

// Wrap your application root once
<ThemeProvider>
  <App />
</ThemeProvider>
```

`ThemeProvider` re-exports `ComponentProps<typeof Provider>`, so any `Provider` prop can be overridden:

```tsx
// Override locale for a localised section
<ThemeProvider locale="de-DE">
  <SettingsPanel />
</ThemeProvider>

// Opt in to a different scale for accessibility
<ThemeProvider scale="large">
  <AccessibleView />
</ThemeProvider>
```

**Implementation** (`ThemeProvider.tsx`):

```tsx
import { Provider } from '@adobe/react-spectrum';
import getiTheme from './theme';

export const ThemeProvider = ({ children, ...rest }: ThemeProviderProps) => (
    <Provider
        locale="en-US"
        theme={getiTheme}
        colorScheme="dark"
        scale="medium"
        id="theme-provider"
        {...rest}
    >
        {children}
    </Provider>
);
```

### Storybook

`ThemeProvider` is registered as a global Storybook decorator in `.storybook/preview.tsx`:

```tsx
const withThemeProvider: Decorator = (Story, context) => (
    <ThemeProvider width={'100%'}>
        <Story {...context.args} />
    </ThemeProvider>
);
```

Every story automatically renders inside the Geti theme — no manual wrapping needed.

## How theme merging works (`theme.ts`)

Spectrum's `Theme` type has five slots, each a `CSSModule` (a map of class-name strings exported by a `.module.css` file):

```ts
type Theme = {
    global: CSSModule;   // applied unconditionally
    dark:   CSSModule;   // applied when colorScheme="dark"
    light:  CSSModule;   // applied when colorScheme="light"
    medium: CSSModule;   // applied at medium scale
    large:  CSSModule;   // applied at large scale
};
```

Spectrum adds the resolved class names to the root DOM node so all `var(--spectrum-*)` declarations inside those selectors are available to every descendant.

Geti uses an **additive merge** strategy — it never replaces Spectrum's default classes:

```ts
// theme.ts
function mergeClasses(defaultObj: CSSModule = {}, customObj: CSSModule = {}): CSSModule {
    const merged = { ...defaultObj };
    for (const key in customObj) {
        if (merged[key]) {
            merged[key] = `${merged[key]} ${customObj[key]}`;  // append Geti class
        } else {
            merged[key] = customObj[key];
        }
    }
    return merged;
}

const getiTheme: Theme = {
    dark:   mergeClasses(defaultTheme.dark,   dark),
    light:  mergeClasses(defaultTheme.light,  light),
    large:  mergeClasses(defaultTheme.large,  large),
    medium: mergeClasses(defaultTheme.medium, medium),
    global: mergeClasses(defaultTheme.global, global),
};
```

Both Spectrum's original class names and Geti's class names end up on the root element. Geti wins via **CSS cascade order** — its declarations are appended later in the string, so same-specificity rules in Geti's file override those in Spectrum's file.

## Component-level CSS modules

When a Spectrum token override isn't granular enough for a specific component, Geti components inject additional classes via **`UNSAFE_className`** — Spectrum's approved escape hatch for consumer CSS.

### The `clsx` merge pattern

Every component that uses `UNSAFE_className` routes it through `clsx` before passing it to Spectrum. This safely composes:
- The component's own internal CSS module class
- A variant class derived from a prop
- Any `UNSAFE_className` supplied by the consumer

```tsx
// ActionButton — compose variant class + consumer class
UNSAFE_className={clsx(getColorVariantClass(colorVariant), UNSAFE_className) || undefined}

// Slider — always inject the filled-state class, merge consumer class
UNSAFE_className={clsx(styles.slider, UNSAFE_className)}

// Switch — always inject the emphasized-state class
UNSAFE_className={clsx(styles.switch, UNSAFE_className)}
```

The `|| undefined` guard strips the attribute entirely when no classes are active, keeping the DOM clean.

### Pattern 1: Prop-driven variant classes (`ActionButton`)

`ActionButton` maps a `colorVariant` prop to CSS module classes for three Geti-specific visual modes:

```css
/* ActionButton.module.css */

/* dark — subtle gray hover (toolbars, dark panels) */
.actionButtonDark {
    border: var(--spectrum-alias-border-size-thin) solid transparent !important;
}
.actionButtonDark:global([class*='is-hovered']) {
    border:           var(--spectrum-alias-border-size-thin) solid var(--spectrum-global-color-gray-300) !important;
    background-color: var(--spectrum-global-color-gray-100) !important;
    border-radius:    var(--spectrum-alias-border-radius-small) !important;
}

/* light — blue-tinted hover (light-background headers) */
.actionButtonLight:global([class*='is-hovered']) {
    border:           var(--spectrum-alias-border-size-thin) solid var(--blue-header-option-hover) !important;
    background-color: var(--blue-header-option-hover) !important;
}

/* blue — transparent with energy-blue text */
.actionButtonBlue:not(:global([class*='disabled'])) {
    color: var(--energy-blue) !important;
}
```

The `:global([class*='is-hovered'])` selector reaches into Spectrum's BEM-style state classes without breaking CSS module encapsulation.

### Pattern 2: Data-attribute-driven tokens (`Slider`, `Switch`)

`Slider` and `Switch` use `data-*` attributes set by the component as CSS hooks, then override Spectrum tokens only in the relevant state:

```css
/* Slider.module.css */
.slider[data-filled='true'] {
    --spectrum-slider-fill-track-color:           var(--energy-blue);
    --spectrum-slider-handle-border-color:        var(--energy-blue);
    --spectrum-slider-handle-border-color-hover:  var(--energy-blue-light);
    --spectrum-slider-handle-border-color-down:   var(--energy-blue-lighter);
}
```

```css
/* Switch.module.css */
.switch[data-emphasized='true'] {
    --spectrum-switch-emphasized-track-color-selected:       var(--energy-blue);
    --spectrum-switch-emphasized-track-color-selected-hover: var(--energy-blue-light);
    --spectrum-switch-emphasized-track-color-selected-down:  var(--energy-blue-lighter);
}
```

This keeps the override scoped to an element state rather than bleeding across all instances.

### Pattern 3: Structural layout overrides (`ToggleButtons`)

When Spectrum's border/radius tokens produce the wrong shape for a composite, structural CSS is applied directly:

```css
/* ToggleButtons.module.css */
.toggleButton:not(:last-child):not(:first-child) {
    border-radius: 0;
    border-right-color: transparent !important;
    border-left-color:  transparent !important;
}
.toggleButton:first-child {
    border-bottom-right-radius: 0;
    border-top-right-radius:    0;
}
.toggleButton:last-child {
    border-bottom-left-radius: 0;
    border-top-left-radius:    0;
}
```

## Token consumption in components

Geti components reference tokens in three ways, in priority order:

### 1. Spectrum style props (preferred)

Spectrum's `View`, `Flex`, `Grid`, and similar layout primitives accept style props that map directly to tokens:

```tsx
<View
    backgroundColor="gray-100"
    borderColor="gray-300"
    padding="size-200"
    borderRadius="medium"
/>
```

### 2. `UNSAFE_className` + CSS Modules

For overrides that Spectrum style props cannot express — interactive state styling, custom hover colours, structural shapes:

```tsx
import styles from './MyComponent.module.css';

<SpectrumWidget UNSAFE_className={clsx(styles.myWidget, UNSAFE_className)} />
```

```css
/* MyComponent.module.css */
.myWidget {
    --spectrum-some-token: var(--energy-blue); /* local token override */
}
```

### 3. Inline `CSSProperties` with `var()` references

For runtime-computed values only (e.g. dynamic colours passed as props):

```tsx
<div
    style={{ backgroundColor: `var(--spectrum-global-color-gray-${shade})` }}
/>
```

> **Do not use `UNSAFE_style`** for static values — CSS modules are always preferable for static overrides because they are properly scoped, cacheable, and do not cause React re-renders.

## Writing custom CSS against Geti tokens

When building a new component or story that needs to reference Geti's brand tokens, use these canonical variable names:

```css
.myElement {
    /* Brand blues */
    color:            var(--energy-blue);
    border-color:     var(--energy-blue-shade);

    /* Gray surfaces */
    background-color: var(--spectrum-global-color-gray-100);
    border:           1px solid var(--spectrum-global-color-gray-300);

    /* Semantic */
    color:            var(--spectrum-global-color-positive);   /* moss green */
    color:            var(--spectrum-global-color-negative);   /* coral red */
    color:            var(--spectrum-global-color-informative);/* energy blue */
    color:            var(--spectrum-global-color-warning);    /* daisy yellow */
}
```

All of the above resolve automatically inside any element that is a descendant of `ThemeProvider`.

## Rules and constraints

| Rule | Reason |
|---|---|
| Always render inside `ThemeProvider` | Token variables are undefined outside the Provider's root element |
| Use `UNSAFE_className` for CSS overrides, not `UNSAFE_style` | `UNSAFE_style` bypasses CSS module scoping and forces inline style recalculation |
| Never use `UNSAFE_className` in tests | Test selectors rely on ARIA roles and text, not class names; class names change with CSS Modules hashing |
| Wrap `UNSAFE_className` through `clsx` | Prevents invalid class strings when values are `undefined`, `false`, or conditionally set |
| Prefer token variables over hard-coded hex values | Hard-coded values don't respond to future theme changes or scale changes |
| Use `!important` only inside `:global([class*='is-*'])` selectors | Necessary to override Spectrum's inline BEM state classes; should never appear in normal scoped rules |

## Adding a new token

1. **Global brand token** — add to `geti-global.module.css` inside `.spectrum { … }`. Prefix with `--brand-`, `--energy-`, or another product namespace.
2. **Dark-mode value** — add to `geti-dark.module.css` inside `.spectrum--dark { … }`.
3. **Spectrum alias override** — add to the appropriate file depending on scope (`.spectrum` for always-active, `.spectrum--dark` for dark-only).
4. **Component-scoped token** — override the specific Spectrum component token in that component's `.module.css` file, scoped behind a class or data attribute.

No manual import or registration is needed — `theme.ts` picks up any exported class name from the module files automatically through `mergeClasses`.

## Migration path

The current stack is **React Spectrum v3** (Spectrum 1 tokens). The planned progression is:

```
Phase 1 (now)  → React Spectrum v3 + CSS module token overrides
Phase 2        → @react-spectrum/s2 (Spectrum 2 tokens, new component APIs)
Phase 3        → react-aria-components + Tailwind CSS (fully headless)
```

At each phase, the token layer described here will need to be re-mapped. The `ThemeProvider` boundary is the isolation point — consuming applications need only update `ThemeProvider` when the token strategy changes, with no changes required to individual component usage.
