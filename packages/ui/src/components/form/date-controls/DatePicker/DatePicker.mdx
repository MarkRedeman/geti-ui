# DatePicker

<Subtitle>A date input that combines a segmented keyboard field with a calendar popover, so users can type or click to pick a date.</Subtitle>

```tsx
import { DatePicker } from '@geti/ui';
```

- [React Spectrum DatePicker](https://react-spectrum.adobe.com/react-spectrum/DatePicker.html)

## Responsibility

`DatePicker` is a thin wrapper around Adobe React Spectrum's `DatePicker`. It applies a single hardcoded `width` default to fix a Safari mobile layout regression before forwarding all props to the Spectrum component.

Key responsibilities:
- **Dual Input Mode**: Offers both a `DateField` (segmented keyboard input for month/day/year) and a calendar button that opens a `Calendar` popover. Users can use whichever method they prefer.
- **Safari Mobile Fix**: Applies `width="size-2400"` before the props spread. Without an explicit width, iOS Safari causes layout flickering when the calendar popover opens. Callers can override this by passing their own `width` (the spread wins).
- **Time Support**: When `granularity="minute"` (or finer) is set, the date field extends to include time segments тАФ no separate `TimePicker` needed.
- **Validation**: Supports `minValue`, `maxValue`, `isDateUnavailable`, `validationState`, and `errorMessage` for comprehensive date constraint expression.

### When to use
- Any form field where a user needs to enter a specific date тАФ scheduled events, deadlines, filter start dates.
- When users may prefer keyboard entry (power users) OR visual calendar selection (casual users) тАФ `DatePicker` supports both.
- When you need combined date-and-time input тАФ set `granularity="minute"` or `granularity="second"`.

### When to avoid
- When only keyboard input is required and no calendar UI is needed тАФ use `DateField` directly.
- When the user needs to select a start **and** end date тАФ use `DateRangePicker` instead.
- When the date selection is from a very large range where calendar navigation would be tedious тАФ consider a plain `TextField` with format guidance.

## Design

Thin wrapper with one hardcoded default prop applied **before** the spread:

```tsx
export const DatePicker = <T extends DateValue>(
  props: SpectrumDatePickerProps<T>
) => (
  <SpectrumDatePicker width="size-2400" {...props} />
);

export type { SpectrumDatePickerProps as DatePickerProps };
```

Because `width="size-2400"` appears before `{...props}`, any `width` the caller provides will override it. No CSS modules, no local state, no custom logic.

## Integration

### Usage examples

```tsx
// Uncontrolled
<DatePicker label="Event date" />

// Controlled with CalendarDate
import { today, getLocalTimeZone } from '@internationalized/date';

const [date, setDate] = useState(today(getLocalTimeZone()));
<DatePicker
  label="Deadline"
  value={date}
  onChange={setDate}
  minValue={today(getLocalTimeZone())}
/>

// With time
<DatePicker
  label="Event date and time"
  granularity="minute"
  onChange={(dateTime) => console.log(dateTime.toString())}
/>

// Validation
<DatePicker
  label="End date"
  isRequired
  errorMessage="A valid end date is required."
  validationState="invalid"
/>
```

- **Depends on**: `@adobe/react-spectrum` тАФ `DatePicker`, `SpectrumDatePickerProps`; `@react-types/datepicker` тАФ `DateValue`
- **Peer imports**: `@internationalized/date` тАФ `CalendarDate`, `today`, `getLocalTimeZone` for constructing typed date values
- **Companion**: `DateRangePicker` (same Safari fix, selects a start + end range); `DateField` (keyboard-only, no calendar popover)
- **Generic**: The `<T extends DateValue>` generic allows using `CalendarDate`, `CalendarDateTime`, or `ZonedDateTime` as the value type тАФ TypeScript will infer this from `value` or `defaultValue`

### Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `value` | `T extends DateValue` | тАФ | The current date (controlled). |
| `defaultValue` | `T extends DateValue` | тАФ | The initial date (uncontrolled). |
| `onChange` | `(value: T) => void` | тАФ | Called when the user commits a date. |
| `label` | `string` | тАФ | The visible label for the field. |
| `granularity` | `'day' \| 'hour' \| 'minute' \| 'second'` | `'day'` | The finest time unit to display. |
| `minValue` | `DateValue` | тАФ | The earliest selectable date. |
| `maxValue` | `DateValue` | тАФ | The latest selectable date. |
| `isDateUnavailable` | `(date: DateValue) => boolean` | тАФ | Function to mark specific dates as unavailable. |
| `isDisabled` | `boolean` | `false` | Prevents interaction. |
| `isReadOnly` | `boolean` | `false` | Displays the value but prevents changes. |
| `isRequired` | `boolean` | `false` | Marks the field as required. |
| `validationState` | `'valid' \| 'invalid'` | тАФ | Controls inline validation styling. |
| `errorMessage` | `ReactNode` | тАФ | Error message shown when `validationState="invalid"`. |
| `width` | `DimensionValue` | `'size-2400'` | Override the default width. |

## Accessibility

- **ARIA role**: The overall component is a `group` containing the segmented date field and a calendar toggle button. Each date segment (month, day, year) is a `spinbutton` тАФ a standard ARIA pattern for discrete numeric inputs with up/down adjustment.
- **Keyboard navigation for field segments**: **Tab** moves between month, day, and year spinbuttons. **Arrow Up/Down** increments or decrements the focused segment. Typing digits directly fills the focused segment and auto-advances to the next.
- **Calendar popover**: The calendar button is labelled `"Calendar"`. When opened, focus moves into the calendar grid. **Arrow keys** navigate between days. **Enter** or **Space** selects the focused date. **Escape** closes the popover and returns focus to the trigger button.
- **Date constraints**: Dates outside `minValue`/`maxValue` or returned unavailable by `isDateUnavailable` are rendered with `aria-disabled="true"` in the calendar grid and cannot be selected by keyboard or pointer.
- **Error announcements**: `errorMessage` is linked to the field via `aria-describedby` and announced by screen readers when validation fails.
- **Labels**: Always provide `label`. If a visible label is not desired, use `aria-label` or `aria-labelledby` to maintain an accessible name for the `group`.

## Related components
- `DateRangePicker` тАФ use instead when users must select both a start and an end date.
- `DateField` тАФ use instead when a calendar popover is not needed and keyboard-only segment input is preferred.
- `Calendar` тАФ use instead when the date picker must be permanently visible rather than in a popover.
