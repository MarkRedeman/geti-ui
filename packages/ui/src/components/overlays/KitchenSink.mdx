## Responsibility

The `overlays/` category owns **every component that renders content above the main UI plane** — modals, popovers, tooltips, and fullscreen takeovers. These components always involve a trigger + floating-panel relationship and manage focus trapping, scroll locking, and portal rendering internally (via Spectrum or react-aria-components).

Components in this folder:

| Component | Purpose |
|---|---|
| `Dialog` / `DialogTrigger` | General-purpose modal dialog (trigger + panel) |
| `AlertDialog` | Pre-structured confirmation/destructive-action dialog |
| `DialogContainer` | Triggerless dialog host for programmatic open/close |
| `Popover` | Lightweight context-anchored popover (thin wrapper over `DialogTrigger type="popover"`) |
| `CustomPopover` | Geti-styled popover built from `react-aria-components` with optional arrow |
| `ContextualHelp` | Small icon-triggered help popover with heading/content |
| `Tooltip` / `TooltipTrigger` | Short hover/focus description attached to any trigger |
| `FullscreenAction` | Inline toolbar button that opens its child content as a fullscreen takeover dialog |

## Design

### Thin Spectrum wrappers (most components)

The standard overlay components are thin wrappers:

```tsx
export const Dialog = (props: DialogProps) => <SpectrumDialog {...props} />;
export const DialogTrigger = (props: DialogTriggerProps) => <SpectrumDialogTrigger {...props} />;
export const Tooltip = (props: TooltipProps) => <SpectrumTooltip {...props} />;
export const TooltipTrigger = (props: TooltipTriggerProps) => <SpectrumTooltipTrigger {...props} />;
```

`Dialog` and `DialogTrigger` are designed to be composed together — `DialogTrigger` holds open state; `Dialog` is the content panel. The split gives callers full control over the dialog content structure (`Heading`, `Content`, `ButtonGroup`, `Divider` are passed as children from Spectrum).

### `Popover` — `DialogTrigger` with defaulted `type`

`Popover` is not a new component but a convenience wrapper that pre-sets `type="popover"` on `DialogTrigger`, so the overlay appears anchored to its trigger rather than as a centred modal:

```tsx
export const Popover = ({ type = 'popover', ...rest }: PopoverProps) => (
  <SpectrumDialogTrigger {...rest} type={type} />
);
```

Callers can override `type` to `'modal'`, `'tray'`, etc. if needed.

### `CustomPopover` — react-aria-components bridge

`CustomPopover` is the one overlay component that uses **`react-aria-components`** rather than `@adobe/react-spectrum`. It wraps RAC's `Popover` + `Dialog` + optional `OverlayArrow`. This gives callers full control over styling (the RAC `Popover` has no pre-applied Spectrum chrome).

Two usage modes:
1. **Controlled** (no `triggerElement`): caller provides open state externally. The component renders the `AriaPopover` directly with no trigger.
2. **Uncontrolled** (with `triggerElement`): the component wraps the trigger and panel in `AriaDialogTrigger`, handling open state automatically.

The optional `showArrow` prop renders an `OverlayArrow` with a simple SVG chevron path.

### `DialogContainer` — triggerless modal pattern

`DialogContainer` is for situations where the element that triggers a dialog unmounts before the dialog closes (e.g., a dialog opened from a list item, then the list re-renders). Spectrum manages focus restoration correctly even without a persistent trigger DOM node.

### `FullscreenAction` — composite Geti component

`FullscreenAction` is the only fully custom composite in this category. It assembles:
- A quiet `ActionButton` with `FullScreen` icon
- A `TooltipTrigger`/`Tooltip` wrapping the trigger
- A `DialogTrigger type="fullscreenTakeover"`
- Inside the dialog: a `Heading`, `Divider`, optional `actionButton` slot, a close button, and a `Content` ref container

The `actionButton` prop can be a static `ReactNode` or a factory function `(ref) => ReactNode` — this allows the action button to receive a ref to the dialog's `Content` container for measurements or canvas operations.

## Flow

1. **Trigger-based overlays** (`DialogTrigger`, `Popover`, `TooltipTrigger`, `FullscreenAction`): trigger element's press/hover/focus opens the overlay; Spectrum manages focus trap + scroll lock internally.
2. **Programmatic overlays** (`DialogContainer`): parent state (`isOpen`) drives visibility; parent calls `onDismiss` to close.
3. **`CustomPopover` uncontrolled**: RAC `DialogTrigger` manages open state entirely internally.
4. **Focus on close**: Spectrum and RAC return focus to the trigger element automatically on close.

---

## Integration

- **Depends on**:
  - `@adobe/react-spectrum` — `Dialog`, `DialogTrigger`, `AlertDialog`, `DialogContainer`, `ContextualHelp`, `Tooltip`, `TooltipTrigger`, `ActionButton`, `ButtonGroup`, `Content`, `Heading`, `Divider`
  - `react-aria-components` — `CustomPopover` (`Popover`, `Dialog`, `DialogTrigger`, `OverlayArrow`)
  - `@spectrum-icons/workflow` — `FullScreen`, `FullScreenExit` icons in `FullscreenAction`
- **Consumed by**: any Geti feature that needs confirmation flows (AlertDialog), contextual options (Popover), help text (ContextualHelp), hover descriptions (Tooltip), or immersive views (FullscreenAction).
- **Theme**: `--spectrum-tray-background-color` and modal overlay colours are overridden in `geti-dark.module.css` to match Geti's dark surface palette.
